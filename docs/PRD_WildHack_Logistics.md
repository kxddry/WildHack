# PRD: Сервис автоматического вызова транспорта на склады

## 1. Контекст и проблема

Склады RWB (Wildberries & Russ) ежедневно отгружают товары по множеству маршрутов. Сейчас вызов транспорта происходит реактивно — когда товар уже готов к отгрузке. Это приводит к:

- Простоям склада (товар готов, машин нет)
- Холостым рейсам (машины приехали, а грузить нечего)
- Росту операционных затрат

**Цель:** Перейти от реактивного к проактивному вызову транспорта на основе прогноза отгрузок.

---

## 2. Решение (high-level)

Веб-сервис, который:

1. Принимает потоковые данные о статусах обработки товаров на складах (8 этапов, обновление каждые 30 мин)
2. Прогнозирует объём отгрузок по каждому маршруту на горизонт 2 часа
3. Рассчитывает необходимое количество транспортных единиц
4. Формирует заявки на вызов транспорта

---

## 3. Пользователи и роли

| Роль | Взаимодействие с системой |
|---|---|
| **Система (автомат)** | Основной потребитель — получает прогнозы и заявки через API |
| **Логист склада** | Просматривает дашборд, может вручную скорректировать заявку |
| **Оператор транспорта** | Получает сформированные заявки на подачу машин |

---

## 4. Данные

### 4.1 Входные данные (из train.parquet)

| Поле | Описание |
|---|---|
| `route_id` | Уникальный ID маршрута |
| `office_from_id` | ID склада-источника (привязка к маршруту фиксированная) |
| `timestamp` | Метка времени (шаг — 30 минут) |
| `status_1..status_8` | Кол-во товаров, прошедших каждый этап обработки за последние 30 мин |
| `target_2h` | **Целевая переменная** — кол-во ёмкостей, отгруженных по маршруту за последние 2 часа |

### 4.2 Тестовые данные

Файл `test.parquet` — те же поля, но без `status_*` и `target_2h`. Нужно предсказать `target_2h`.

### 4.3 Ключевые наблюдения

- Данные временные ряды с шагом 30 мин
- Статусы — это «воронка» обработки товара на складе (status_1 → status_8 → отгрузка)
- Один склад обслуживает несколько маршрутов
- Привязка маршрут↔склад не меняется

## 7. Бизнес-логика Transport Planner

### 7.1 Допущения (фиксируем в README)

| Допущение | Значение |
|---|---|
| Вместимость одной машины | Фиксированная (например, `V = 20` ёмкостей) |
| Время подачи транспорта | 1 час от момента заявки до прибытия на склад |
| Горизонт прогноза | 2 часа (как в целевой переменной) |
| Периодичность пересчёта | Каждые 30 минут (совпадает с шагом данных) |
| Округление кол-ва машин | Вверх (ceil) — лучше пустая машина, чем неотгруженный товар |

### 7.2 Формула расчёта

```
vehicles_required = ceil(sum(predicted_target_2h для всех маршрутов склада) / V)
```

### 7.3 Логика буферизации

- Агрегация прогнозов по складу (один склад → несколько маршрутов → одна заявка)
- Если `vehicles_required == 0` → заявка не формируется
- Если разница с предыдущей заявкой < порога (например, ±1 машина) → не обновлять, чтобы избежать флаппинга

## 8. ML-модель (Forecast Service)

### 8.1 Подход

- **Задача:** Регрессия — предсказать `target_2h` (непрерывное, ≥ 0)
- **Гранулярность:** Для каждой пары (route_id, timestamp)
- **Фичи:** status_1..status_8, временные фичи (час дня, день недели, выходной), лаговые фичи, агрегаты по складу

### 8.2 Кандидаты моделей

| Модель | Почему |
|---|---|
| LightGBM / CatBoost | Быстрый инференс, хорошо работает с табличными данными, легко деплоить |
| Linear Regression | Бейзлайн для сравнения |

### 8.3 Валидация

- TimeSeriesSplit (не случайный split — данные временные)
- Метрика: WAPE + |Relative Bias| (как в соревновании)

### 8.4 Инференс

- Модель сериализуется в `model.pkl` (или `model.cbm`)
- При старте Forecast Service загружает модель в память
- Инференс — CPU-only, без GPU

## 9. Метрики качества системы

### 9.1 ML-метрики

| Метрика | Описание |
|---|---|
| **WAPE** | Взвешенная абсолютная ошибка: `sum(|y - ŷ|) / sum(y)` |
| **Relative Bias** | Систематическое смещение: `|sum(ŷ)/sum(y) - 1|` |
| **WAPE + \|RBias\|** | Финальная метрика соревнования (чем ниже, тем лучше) |

### 9.2 Бизнес-метрики (для презентации)

| Метрика | Описание |
|---|---|
| **Точность заказа транспорта** | % случаев, когда заказанное кол-во машин совпало с фактически нужным (±2) |
| **Простой склада** | Время между «товар готов» и «машина прибыла» |
| **Холостой пробег** | % машин, уехавших загруженными < 50% вместимости |
| **Экономия vs бейзлайн** | Сравнение с наивным подходом (например, среднее за прошлую неделю) |

### 9.3 Бейзлайн для сравнения

- **Наивный:** `predicted = среднее target_2h по (route_id, час_дня, день_недели)` за обучающий период
- **Текущая практика:** Если известна — сравнить с ней

## 13. Пути развития (для презентации)

1. **Обратная связь:** Добавить фактические данные об отгрузках → автоматический ретрейн модели
2. **Дополнительные данные:** Погода, праздники, акции/распродажи, исторические задержки транспорта
3. **Оптимизация маршрутов:** Объединение заявок с соседних складов для экономии на логистике
4. **Адаптивная вместимость:** Разные типы машин (малый/средний/крупный) → задача оптимизации
5. **Real-time pipeline:** Kafka/NATS для потоковой обработки вместо REST-опроса
6. **Мониторинг дрифта модели:** Автоматическое обнаружение деградации прогноза

## 14. Риски

| Риск | Митигация |
|---|---|
| Модель плохо прогнозирует пиковые нагрузки | Добавить фичи: день недели, праздники, лаги по складу |
| Переобучение на train | Строгий TimeSeriesSplit, мониторинг WAPE на валидации |
| Latency инференса > допустимого | Батчинг запросов, кэширование, pre-compute по расписанию |
| Тестовые данные без статусов | Использовать лаговые/временные фичи; возможно, модель должна работать и без status_* |
